Detailed Description

Encapsulates the implementation details and data members of ArgppBase.

Uses the Pimpl idiom in order to facilitate binary compatibility with future releases.

bool argpp::impl::ArgppBaseImpl::isOptionEnd ( const argp_option *  opt)
staticprivate

Called by addOptions() to detect the null delimiting option (an entry with zero in all fields).

See http://www.gnu.org/software/libc/manual/html_node/Argp-Option-Vectors.html#Argp-Option-Vectors for more details. Note that the header argp.h declares a similar function __option_is_end(). We cannot use it, however, because most of the time a linker error occurs if argp is a system library.

Parameters
opt: See the definition of struct argp_option in the header argp.h for more details.
Returns
Returns true if all fields of opt are NULL or 0.

error_t argpp::impl::ArgppBaseImpl::parserRouter ( int  key,
char *  arg,
argp_state *  state 
)
staticprivate

This is the only parser function used in argp++ which communicates with argp as a callback function.

Normally, each parser and child parser supply a pointer to their own parser functions which are called by argp parsing code in turn. However, we use the child_inputs[] vector to store the "this" pointers to the ArgppBase-derived children class instances. This eliminates the necessity of defining multiple parser functions. Each child class merely reimplements the virtual functions of the base class which they need.

Please refer to the code and comments in the argppbaseimpl.cpp file for more details.

Parameters

bool argpp::impl::ArgppBaseImpl::argpp_should_exit_ = false
staticprivate

Determines whether or not argp++ should call exit() after one of the helper functions was called.

Unfortunately, argp does not clean up all of its memory when exit() is called. The most typical use case here is when the user enters "--help", "--version" or "--usage" on the command line. In such cases, the valgrind memcheck tool reports one block not freed but "still reachable" which means that after exit() is called, the OS will free any such memory left dangling by the application. This is not a problem in C code; however, in C++ applications, care must be taken to call destructors of objects which allocate memory in their constructors. This can be difficult to manage, especially where temporary objects are involved.

Therefore, all of the wrappers for the argp helper functions which take references to std::string as arguments are implemented by copying the argp_state data, passing the flags together with ARGP_NO_EXIT, then calling the native helper function, and funally by setting the flag argpp_should_exit. Since the helper function wrappers are invariably called during one of the parser routines, the code in the static parserRouter() function checks this flag after the routine returns and calls exit() if it is set. That way, we can be assured that all of the temporary string objects have been cleaned up before exit() is called.

Clients can pass ARGP_NO_EXIT as well, in which case this flag is always false.
